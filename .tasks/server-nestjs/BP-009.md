# BP-009 — `_*.yaml` shared-roles files treated as invalid blueprints

**Type:** bug
**Severity:** low — cosmetic, but surfaces on every blueprint run
**Affects:** `@rhino-dev/rhino-nestjs@0.1.0`

## Symptom

Laravel convention (shared across all Rhino stacks) is `_roles.yaml` at the root of `.rhino/blueprints/` — a file declaring shared role anchors referenced by model YAMLs via `&anchor` / `*anchor`.

Running `npx rhino blueprint`:

```
✗ _roles.yaml: Invalid blueprint file: missing 'model' key in .../.rhino/blueprints/_roles.yaml
↻ comments.yaml (Comment)
↻ labels.yaml (Label)
↻ projects.yaml (Project)
↻ tasks.yaml (Task)

Blueprint run complete: 4 processed, 0 skipped, 1 errors.
```

The run completes but exits with errors because of an expected shared-anchors file that has no `model:`.

## Root cause

`src/blueprint/blueprint-runner.ts` (library):

```ts
const yamlFiles = fs.readdirSync(blueprintsDir)
  .filter((f) => f.endsWith('.yaml') || f.endsWith('.yml'));
```

It iterates ALL `.yaml` files. No convention to skip leading-underscore files.

## Workaround

Template moved `_roles.yaml` outside the blueprints folder (`.rhino/_roles.yaml`). Works but defeats the YAML anchor feature entirely — the model YAMLs can no longer reference shared anchors via the `_roles.yaml` definitions.

## Proposed fix

```ts
const yamlFiles = fs.readdirSync(blueprintsDir)
  .filter((f) => (f.endsWith('.yaml') || f.endsWith('.yml')) && !f.startsWith('_'));
```

YAML anchors resolve at the parser level regardless of file order. If a consumer wants shared anchors, they can:
1. Merge all `.yaml` files into one document before parsing (YAML anchors only work within a single document).
2. OR process `_roles.yaml` first, extract its top-level map, and pass it to other files via `js-yaml` schema.

The safer change: treat `_*.yaml` as "partial / reference" files and SKIP them in the file discovery loop — don't parse, don't error. Document that the consumer is responsible for manually merging anchors if they need cross-file references.

## Acceptance criteria for fix

- [ ] `_roles.yaml`, `_shared.yaml`, `_common.yaml` etc. are silently skipped.
- [ ] New test: blueprints directory with `_common.yaml` + `posts.yaml` → only `posts.yaml` processed, no error reported.
- [ ] Document behavior in the blueprint docs.
