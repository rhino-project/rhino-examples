# BP-008 — No JSON-array column hydration for dbs without native arrays (SQLite/MySQL)

**Type:** missing feature
**Severity:** high — permissions are unusable on SQLite / MySQL without workaround
**Affects:** `@rhino-dev/rhino-nestjs@0.1.0`

## Problem

The PRD stores `user_roles.permissions` as a JSON array: `["projects.index", "tasks.store", ...]`. Postgres supports `Json` columns natively; SQLite and older MySQL don't. On SQLite, Prisma types `permissions` as `String`.

When `JwtAuthGuard` loads the user:

```ts
const user = await delegate.findUnique({
  where: { id: payload.sub },
  include: { userRoles: { include: { role: true } } },
});
req.user = user;
```

`user.userRoles[0].permissions` comes back as the literal string `'["projects.index","projects.show"]'`, not an array.

`ResourcePolicyGuard` → `userHasPermission(user, 'projects.store', org)` → `resolveUserPermissions(user, orgId)`:

```ts
for (const ur of userRoles) {
  if (orgId === organizationId) {
    const perms = ur.permissions ?? [];
    for (const p of perms) all.push(p);  // ← iterates chars when perms is a string
  }
}
```

Iterating a string yields characters. `'[' !== 'projects.store'` → match never succeeds → 403 on every request for non-admin users.

## Symptom

Dave (viewer in Acme, permissions `["projects.index","projects.show", ...]`):

```bash
curl /api/acme/projects -H "Authorization: Bearer $DAVE_TOKEN"
→ {"code":"FORBIDDEN","message":"This action is unauthorized."}
```

## Workaround

Template wraps PrismaClient with `$extends` → `$allOperations` → walk the result tree → parse any object's `permissions` string into an array. See `src/prisma-extensions.ts`.

```ts
export function createExtendedPrisma(): any {
  return new PrismaClient().$extends({
    query: {
      $allModels: {
        async $allOperations({ operation, args, query }) {
          const result = await query(args);
          if (['findMany','findFirst','findUnique','findFirstOrThrow','findUniqueOrThrow'].includes(operation)) {
            return walk(result);
          }
          return result;
        },
      },
    },
  });
}
```

Works, but:
- Walks every returned tree once per query — cost on big payloads.
- Only handles fields named `permissions` — brittle.
- Consumers must remember to wire this.

## Proposed fix

### Option A — Library-level support via config
Add to `RhinoConfig`:

```ts
auth: {
  permissionsField: 'permissions',     // column name on UserRole
  permissionsFormat: 'json-string' | 'array' | 'comma-separated',
}
```

Then in `permission-matcher.ts`:
```ts
function coercePermissions(raw: unknown): string[] {
  if (Array.isArray(raw)) return raw as string[];
  if (typeof raw === 'string') {
    try {
      const parsed = JSON.parse(raw);
      return Array.isArray(parsed) ? parsed : [];
    } catch {
      return raw.split(',').map(s => s.trim()).filter(Boolean);
    }
  }
  return [];
}
```

Wrap every `ur.permissions ?? []` with `coercePermissions(ur.permissions)` inside `resolveUserPermissions`. Zero consumer wiring.

### Option B — Ship a PrismaClient extension helper

Export `withJsonPermissions(prisma)` from the library. Consumers call it when wiring up.

Recommendation: **Option A**. The JSON-string case is extremely common on SQLite and MySQL 5.7. Making the matcher robust costs ~5 lines.
