# BP-005 — Blueprint YAML ignores `has_uuid: true` option

**Type:** missing feature
**Severity:** medium — UUID-PK models (PRD: `Comment`) can't be generated
**Affects:** `@rhino-dev/rhino-nestjs@0.1.0`

## Problem

Adding `options.has_uuid: true` to a blueprint YAML:

```yaml
model: Comment
options:
  has_uuid: true
  soft_deletes: true
```

...changes NOTHING in the generated output:

- `prisma/schema.prisma` still gets `id Int @id @default(autoincrement())` (not `String @id @default(uuid())`).
- `ProjectResource.ts` gets no `hasUuid: true` flag on the registration.
- Seeder generates sequential integer ids.
- Test generator emits `id: 1` sample rows (integer) instead of UUID strings.

## Root cause

`src/blueprint/blueprint-parser.ts` parses only `belongs_to_organization`, `soft_deletes`, `audit_trail`, `owner`, `except_actions`, `pagination`, `per_page` from `options:`. The `has_uuid` key is silently dropped.

`src/blueprint/generators/*`:
- `prisma-schema-generator.ts` — hard-codes the `id` column.
- `resource-definition-generator.ts` — never emits `hasUuid: true`.
- `seeder-generator.ts` — assumes integer primary keys when picking seed `id: 1, 2, 3`.
- `test-generator.ts` — `buildSampleRow()` hard-codes `id: 1`.

## Workaround

In `src/rhino.config.ts` the template spreads `...commentsRegistration` then adds `hasUuid: true` manually. The `id` column in `prisma/schema.prisma` was hand-authored as `String @id @default(uuid())`.

## Proposed fix

1. `blueprint-parser.ts` — add `has_uuid?: boolean` to `BlueprintOptions`.
2. `prisma-schema-generator.ts` — when `has_uuid: true`, emit `id String @id @default(uuid())` instead of the Int PK line.
3. `resource-definition-generator.ts` — append `hasUuid: true,` to `ModelRegistration` when set.
4. `seeder-generator.ts` — when `has_uuid`, seed with `id: randomUUID()` (import from `crypto`).
5. `test-generator.ts` — sample row id as UUID string when `has_uuid`.
6. `blueprint-validator.ts` — when `has_uuid` + `belongs_to_organization`, validate the model has `organizationId` foreign column declared.

## Acceptance criteria for fix

- [ ] Generator emits `id String @id @default(uuid())` when `has_uuid: true`.
- [ ] Generated Comment.spec correctly uses UUID strings in assertions.
- [ ] Consumer's `ModelRegistration.hasUuid` is `true` without hand-editing.
- [ ] Library test: blueprint with `has_uuid: true` round-trips through all 5 generators with coherent types.
