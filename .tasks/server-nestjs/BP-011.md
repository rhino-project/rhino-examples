# BP-011 — Cross-tenant access returns 403 instead of 404

**Type:** parity divergence
**Severity:** medium — breaks PRD AC-1 expectation of 404
**Affects:** `@rhino-dev/rhino-nestjs@0.1.0`

## Symptom

PRD §6 AC-1:
> Cannot reference Acme's project_id when creating a task in Globex — cross-tenant access returns 404 (Organization not found).

Laravel impl returns `404` when Alice (Acme user) hits `/api/globex/projects`. NestJS impl returns `403 FORBIDDEN`.

```bash
curl /api/globex/projects -H "Authorization: Bearer $ALICE_TOKEN"
→ {"code":"FORBIDDEN","message":"This action is unauthorized."}

# Laravel equivalent:
→ {"message":"Organization not found","status":404}
```

## Root cause

In the template's workaround `tenantRewrite` middleware (`src/main.ts`):

```ts
let org = orgCache.get(first);
if (org === undefined) {
  org = (await tenantPrisma.organization.findUnique({ where: { slug: first } })) ?? null;
  orgCache.set(first, org);
}
if (!org) return next();  // ← doesn't throw 404; lets request continue
```

When the org slug exists but the user isn't a member, the middleware also falls through. Then `GlobalController` sees `req.organization = undefined`, `ResourcePolicyGuard` checks permissions without tenant context, Alice's tenant-keyed permissions don't apply → `userHasPermission(..., null)` falls back to `user.permissions` (null) → denied with 403.

If the org slug literally doesn't exist, we fall through to NestJS routing which treats `globex` as `:modelSlug` → `UNKNOWN_RESOURCE` 404.

Both are wrong versus Laravel spec.

## Fix (library-side, AFTER BP-001)

Once route groups register actual routes, `ResolveOrganizationMiddleware` runs properly. That middleware ALREADY throws `NotFoundException('Organization not found')` in both cases:

```ts
async use(req, _res, next) {
  const identifier = req.params.organization ?? req.params.org;
  if (!identifier) return next();
  const org = await this.orgService.resolve(identifier, req.user);
  req.organization = org;
  next();
}
```

And `OrganizationService.resolve`:

```ts
if (!org) throw new NotFoundException('Organization not found');
if (user) await this.ensureMembership(user, org);  // also throws NotFoundException
```

So: **once BP-001 is fixed, BP-011 is fixed for free.** The current 403 is a symptom of the same underlying routing gap.

## Workaround for the template today

Strengthen `tenantRewrite` to throw 404 when:
1. The URL first segment is NOT a reserved path (`auth`, `invitations`, `nested`), AND
2. The first segment looks like a tenant slug (not a known model slug), AND
3. EITHER the org doesn't exist OR the user isn't a member.

```ts
async function tenantRewrite(req, res, next) {
  const m = raw.match(/^\/api\/([^/]+)(\/.*)?$/);
  if (!m) return next();
  const first = m[1];
  if (['auth','invitations','nested'].includes(first)) return next();
  if (KNOWN_MODEL_SLUGS.has(first)) return next();  // not a tenant URL

  const org = await resolveOrg(first);
  if (!org) { res.status(404).json({ code: 'NOT_FOUND', message: 'Organization not found' }); return; }

  if (req.user) {
    const member = await prisma.userRole.findFirst({ where: { userId: req.user.id, organizationId: org.id } });
    if (!member) { res.status(404).json({ code: 'NOT_FOUND', message: 'Organization not found' }); return; }
  }

  req.organization = org;
  req.url = `/api${m[2] ?? ''}`;
  next();
}
```

**Not currently implemented in the template** — `rhino-template/server-nestjs` returns 403 today. Fix this when BP-001 lands so the workaround can be deleted entirely.

## Acceptance criteria for fix

- [ ] Cross-tenant access (user NOT a member of org) → 404 `{code: 'NOT_FOUND', message: 'Organization not found'}`.
- [ ] Non-existent org slug → 404 same shape.
- [ ] Reserved paths (`/api/auth/*`, `/api/invitations/accept`, `/api/nested`) unaffected.
- [ ] E2E parity test: same request produces same status on Laravel (8001) and NestJS (8004).
