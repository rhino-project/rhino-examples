# BP-007 — `SerializerService` calls policy attribute methods without `org` context

**Type:** bug
**Severity:** critical — every generated policy with `hasRole(user, role, org)` is dead code in serializer path
**Affects:** `@rhino-dev/rhino-nestjs@0.1.0`

## Symptom

Alice is seeded as `admin` in Acme. The generated `ProjectPolicy.permittedAttributesForShow` says:

```ts
override permittedAttributesForShow(user: any, org?: any): string[] {
  if (this.hasRole(user, 'owner', org) || this.hasRole(user, 'admin', org)) {
    return ['*'];
  }
  // ... fallback returns []
}
```

Alice hits `GET /api/acme/projects`. Response:

```json
{"data":[{"id":1},{"id":2},{"id":3}]}
```

Only `id` — every other column stripped. The whitelist collapsed to `[]`, meaning the admin branch did NOT match.

## Root cause

`src/services/serializer.service.ts` (library):

```ts
serializeOne(record, reg, user) {
  // ...
  if (reg.policy) {
    const policy = new reg.policy();
    const hidden = policy.hiddenAttributesForShow(user) ?? [];   // ← no org
    for (const col of hidden) delete result[col];
    const permitted = policy.permittedAttributesForShow(user) ?? ['*'];  // ← no org
    if (!(permitted.length === 1 && permitted[0] === '*')) {
      const keep = new Set([...permitted, 'id']);
      result = Object.fromEntries(Object.entries(result).filter(([k]) => keep.has(k))) as any;
    }
  }
}
```

`org` never passed. Base class `hasRole(user, role, organization?)`:

```ts
protected hasRole(user: any, role: string, organization?: any): boolean {
  if (!user) return false;
  const orgId = organization?.id;
  if (orgId == null) return false;   // ← always false when org is undefined
  return resolveUserRoleSlug(user, orgId) === role;
}
```

Result: `hasRole(user, 'admin', undefined)` → `orgId === undefined` → returns false. Every role check in every generated policy fails. All responses get `permittedAttributes: []` → `{id: N}` only.

## Proof

Change the generated policy to NOT use `org`:

```ts
if (user.userRoles[0]?.role?.slug === 'admin') return ['*'];
```

Suddenly admin sees everything. Proves the `org` path is dead.

## Workaround

`src/policies/ActiveRolePolicy.ts` in the template overrides `hasRole` to resolve role from `user.userRoles[0].role.slug` directly (no org required). Every generated `*Policy.ts` is rewritten to `extends ActiveRolePolicy`.

Breaks multi-org users, but every seeded user belongs to exactly one org.

## Proposed fix

Three options, in order of preference:

### Option A — Thread `org` through the serializer call chain

Every caller of `serializeOne` / `serializeMany` already has `req.organization` in scope. Change:

```ts
- serializeOne(record, reg, user)
+ serializeOne(record, reg, { user, organization })
```

Update `GlobalController.index/show/store/update/trashed/restore` + `HiddenColumnsInterceptor` to pass the org.

**Pros:** clean, no state.
**Cons:** breaking change to `SerializerService` signature.

### Option B — Request-scoped policy context

Use NestJS's `@Inject(REQUEST)` in `SerializerService` (make it `@Injectable({ scope: Scope.REQUEST })`). Read org from the request inside the service.

**Pros:** no controller-side changes.
**Cons:** request-scoped services have performance cost (Nest re-instantiates per-request).

### Option C — Store "active org" on `user`

Convention: whoever loads `req.user` also sets `user.__activeOrg = req.organization`. Base `ResourcePolicy.hasRole` reads it from there if `organization` arg is missing.

**Pros:** no signature change.
**Cons:** hidden state, surprises consumers.

**Recommendation: Option A.** It's a 0.2.0 major bump — call it out in the changelog.

## Acceptance criteria for fix

- [ ] `serializeOne(record, reg, { user, organization })` signature.
- [ ] `ValidationService` already receives `{ action, user, organization }` → consistent.
- [ ] `GlobalController` passes `req.organization` to every serializer call.
- [ ] `HiddenColumnsInterceptor` passes `req.organization` too.
- [ ] E2E: Alice (admin in Acme) `GET /api/acme/projects/1` sees every field.
- [ ] E2E: Dave (viewer in Acme) `GET /api/acme/projects/1` sees only `permittedAttributesForShow('viewer')` fields.
- [ ] Update generator output — remove need for `ActiveRolePolicy` workaround.
