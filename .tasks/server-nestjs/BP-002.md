# BP-002 — `PrismaSchemaGenerator` appends duplicate models with broken field shape

**Type:** bug
**Severity:** high — generated Prisma schema fragment is not usable as-is
**Affects:** `@rhino-dev/rhino-nestjs@0.1.0`

## Symptom

Running `npx rhino blueprint` unconditionally appends new `model <Name> { ... }` blocks to `prisma/schema.prisma`. Two problems:

1. **No detection of existing models.** If the consumer already hand-authored the model (the expected flow for non-trivial domains — FK chains, polymorphic relations, non-Prisma-default IDs), the generator appends a second `model Foo` declaration. Prisma refuses to run (duplicate model error).

2. **The appended models are not production-usable:**
   - No `@relation` blocks for FKs: generator emits `projectId Int` but not `project Project @relation(fields: [projectId], references: [id])`, so `prisma.task.findMany({ include: { project: true } })` throws at type-gen time.
   - No inverse relation on parent models.
   - `organizationId` column is emitted for `belongs_to_organization: true` but NO relation.
   - `has_uuid: true` is ignored — Comment still gets `id Int @id @default(autoincrement())`.
   - `@@unique([...])` and `@@index([...])` directives are not generated.
   - `@default()` values for `enum` / `string` types are emitted inconsistently (duplicate spaces in output).
   - `decimal` type maps to `Decimal?` but Prisma SQLite driver doesn't support `Decimal` — causes runtime failure.

## Actual output (from copying Laravel `projects.yaml`)

```prisma
model Project {
  id        Int      @id @default(autoincrement())
  organizationId Int
  organization   Organization @relation(fields: [organizationId], references: [id])
  title        String    
  description  String?   
  status       String     @default("draft")
  budget       Decimal?    // ← breaks SQLite
  internal_notes String?   // ← snake_case (see BP-003)
  starts_at    DateTime? 
  ends_at      DateTime? 
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
  deletedAt DateTime?

  @@map("projects")
}

model Task {
  id        Int      @id @default(autoincrement())
  title        String    
  description  String?   
  status       String     @default("todo")
  priority     String     @default("medium")
  estimated_hours Decimal?  
  due_date     DateTime? 
  project_id   Int         // ← no @relation → unusable
  assignee_id  Int?        // ← no @relation
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
  deletedAt DateTime?

  @@map("tasks")
}
```

Task above has a raw `project_id Int` with no relation — the consumer cannot `prisma.task.create({ data: { project: { connect: {...} } } })` or include the related project.

## Root cause

`src/blueprint/generators/prisma-schema-generator.ts` (library):

- Walks `blueprint.columns` and emits `<name> <type>` lines.
- Has no awareness of `relationships:` inverse wiring.
- No check for an existing file+model; always appends.
- `@relation` is only emitted for `organization` (because `belongs_to_organization: true` hard-codes that one case).
- `foreignId` column type maps to `Int` but drops the relation entirely.
- No handling of `has_uuid: true` on the model options.

## Repro

```bash
cd rhino-template/server-nestjs
# With a clean Prisma schema (only Organization, User, UserRole, Role)
rm prisma/schema.prisma
cat > prisma/schema.prisma <<'EOF'
generator client { provider = "prisma-client-js" }
datasource db { provider = "sqlite"; url = env("DATABASE_URL") }
model Organization { id Int @id @default(autoincrement()); slug String @unique }
EOF
npx rhino blueprint
# Inspect prisma/schema.prisma — appended models lack @relation, have Decimal, etc.
```

## Workaround

The template **hand-authors `prisma/schema.prisma`** with proper relations, UUIDs, timestamps, and soft-delete columns. `npx rhino blueprint` still appends garbage; a post-run step truncates the file back.

## Proposed fix

### Short term (v0.1.x)
- Add `--skip-prisma` CLI flag. Generator writes `resource / policy / test / seeder` only.
- Warn loudly when a model named in a blueprint already exists in `prisma/schema.prisma` and skip rewriting it (respect hand-authored overrides).

### Medium term (v0.2.0)
Rewrite `PrismaSchemaGenerator` to:
1. Parse the existing `prisma/schema.prisma` into an AST (use `@mrleebo/prisma-ast` or similar).
2. Upsert the target model — preserving hand-authored fields not in the blueprint.
3. Emit proper `@relation(fields: [...], references: [...])` for every FK declared in `relationships:`.
4. Emit the inverse side on the parent (requires the blueprint of the parent model OR a second pass across all blueprints).
5. Map `foreignId` → `Int @relation(...)` together — never bare `Int`.
6. Map `decimal` → `Float` when `datasource.provider == sqlite`, `Decimal` otherwise.
7. Honor `has_uuid: true` → `id String @id @default(uuid())`.
8. Emit `@@unique` / `@@index` from a new `indexes:` YAML section.
9. Emit timestamps (`createdAt`, `updatedAt`) only when `options.timestamps` is `true` (default true), not always.

### Blueprint YAML extension
```yaml
model: Comment
options:
  has_uuid: true    # → id String @id @default(uuid())
  timestamps: true
  soft_deletes: true

indexes:
  - [taskId, userId]
  - taskId

relationships:
  - type: belongsTo
    model: Task
    foreign_key: taskId   # column on THIS model
    # inverse is inferred if target blueprint has `hasMany: Comment`
```

## Acceptance criteria for fix

- [ ] Running blueprint twice on the same directory produces no duplicate models.
- [ ] Generated Prisma schema compiles with `prisma generate` and migrates with `prisma migrate dev`.
- [ ] `prisma.task.findMany({ include: { project: true } })` works on a freshly-generated schema.
- [ ] `prisma.comment.findMany({ include: { task: { include: { project: true } } } })` works (3-level relation from blueprints).
- [ ] Generated schema uses `Float` on SQLite and `Decimal` on Postgres.
- [ ] Comment with `has_uuid: true` has `id String @id @default(uuid())`.
- [ ] E2E test: run blueprint against 5 interdependent YAMLs (org, project, task, comment, label) → migrate → hit every endpoint.
