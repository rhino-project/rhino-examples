# BP-004 — Blueprint YAML has no way to declare `fkConstraints` / indirect tenancy (`owner:`)

**Type:** missing feature
**Severity:** high — cross-tenant FK validation (PRD AC-8) is impossible from blueprint alone
**Affects:** `@rhino-dev/rhino-nestjs@0.1.0`

## Problem

The library supports two features that cannot be expressed in a blueprint YAML:

1. `ModelRegistration.fkConstraints: Array<{ field, model }>` — tenant-safe FK validation. Used by `ValidationService.verifyTenantFks()` to confirm `payload.projectId` points to a project in the same org.

2. `ModelRegistration.owner: string` — names the relation that leads to the organization. E.g. `Task.owner = 'project'` (indirect tenancy; `Task` has no `organizationId`).

Both are needed by the PRD's Task and Comment models (indirect tenants). Neither has YAML syntax.

## Current workaround

In `src/rhino.config.ts`:

```ts
tasks: {
  ...tasksRegistration,
  owner: 'project',
  fkConstraints: [{ field: 'projectId', model: 'project' }],
  ...
},
comments: {
  ...commentsRegistration,
  owner: 'task.project',   // ← nested walk
  fkConstraints: [{ field: 'taskId', model: 'task' }],
  ...
},
```

Every consumer must repeat this — the generator can't do it from the YAML.

## Proposed YAML extension

```yaml
model: Task
options:
  belongs_to_organization: false
  # Declare indirect tenancy. Walker: Task → Project → Organization
  owner_chain:
    - projectId: project          # via Task.projectId → Project
    # stops here — Project has belongs_to_organization: true

fk_constraints:
  - field: projectId
    model: project
  - field: assignedTo
    model: user
    # Users aren't tenant-scoped, but we can still check existence
    check: exists
```

Generator should emit:

```ts
export const tasksRegistration: ModelRegistration = {
  // ...
  owner: 'project',
  fkConstraints: [
    { field: 'projectId', model: 'project' },
    { field: 'assignedTo', model: 'user' },
  ],
};
```

## Root-cause location

- `src/blueprint/blueprint-parser.ts` — add `owner_chain` and `fk_constraints` keys to `BlueprintOptions` and `Blueprint` types.
- `src/blueprint/blueprint-validator.ts` — validate that every `fk_constraints[].model` maps to another blueprint in the set.
- `src/blueprint/generators/resource-definition-generator.ts` — emit the extra lines.
- `src/services/validation.service.ts` (already supports `fkConstraints` — just needs the data).

## Acceptance criteria for fix

- [ ] YAML can declare `fk_constraints: [{ field, model }]` → emitted verbatim to `ModelRegistration`.
- [ ] YAML can declare `owner_chain` → emitted as `owner` string.
- [ ] Validator errors when an FK points to an unknown blueprint.
- [ ] E2E test: Task blueprint with `fk_constraints: [{ field: projectId, model: project }]` + cross-tenant POST returns `{ code: 'CROSS_TENANT' }`.
- [ ] Consumer template can remove the hand-authored overrides in `rhino.config.ts`.
