# BP-001 — Route-group `prefix: ':organization'` never registers tenant routes

**Type:** bug
**Severity:** critical — multi-tenant apps are unusable without a workaround
**Affects:** `@rhino-dev/rhino-nestjs@0.1.0`

## Symptom

With the documented config:

```ts
RhinoModule.forRoot({
  prismaClient,
  models: { projects: { model: 'project', belongsToOrganization: true } },
  routeGroups: {
    tenant: { prefix: ':organization', models: '*' },
  },
  multiTenant: { organizationIdentifierColumn: 'slug' },
})
```

A GET to `/api/acme/projects` returns:

```json
{ "code": "UNKNOWN_RESOURCE", "message": "Unknown resource: acme", "details": { "slug": "acme" } }
```

Because NestJS only sees the routes declared by `@Controller()` decorators on the library's controllers:

```
GET /api/:modelSlug
GET /api/:modelSlug/:id
POST /api/:modelSlug
...
```

`acme` is matched as `:modelSlug` and the config has no `acme` model.

## Root cause

`RhinoModule` stores `routeGroups` in the config and exposes them via `RhinoConfigService`, but never uses them to actually register routes. `GlobalController`, `AuthController`, `InvitationController`, and `NestedController` all have hard-coded `@Controller()` paths. NestJS cannot dynamically add prefixes to an existing controller at runtime.

Looking at `src/rhino.module.ts` (library):
- `configure()` applies `RouteGroupMiddleware` and `ResolveOrganizationMiddleware` globally
- Neither changes the registered route table
- `ResolveOrganizationMiddleware` expects `req.params.organization`, but with `@Controller('api/:modelSlug')` the `:organization` segment is never a param — the URL literally doesn't route through that controller

## Repro

Inside `rhino-template/server-nestjs`:

1. `npm ci && npx prisma migrate dev && npx ts-node prisma/seed.ts`
2. `npx ts-node src/main.ts`
3. `TOKEN=$(curl -s -XPOST http://localhost:8004/api/auth/login -H 'Content-Type: application/json' -d '{"email":"alice@acme.com","password":"password123"}' | jq -r .token)`
4. `curl http://localhost:8004/api/acme/projects -H "Authorization: Bearer $TOKEN"` → 404 `UNKNOWN_RESOURCE`

## Workaround (currently in use)

`src/main.ts` installs a raw-Express middleware via `app.use()` BEFORE Nest's router runs. It rewrites `/api/<slug>/...` → `/api/...` when `<slug>` matches an organization, setting `req.organization` in the process.

This works but:
- Bypasses the library's `ResolveOrganizationMiddleware` entirely
- Requires a cold DB lookup per unknown slug (cached after first hit)
- Loses the per-group `middleware` and `skipAuth` features
- Consumers must remember to register it before `app.listen()`

## Proposed fix

Three candidate approaches (preference in order):

### Option A — Dynamic controller registration

For each route group, generate a NEW controller class at module-init time whose `@Controller()` path is the resolved prefix, then register all matching model slugs inside it. Nest supports this via `DynamicModule.controllers` returned from `forRoot`, but the controller class must be generated dynamically with `Reflect.defineMetadata(PATH_METADATA, prefix, class)`.

**Pros:** full parity with Laravel, no raw-Express escape hatch.
**Cons:** complex reflection, harder to introspect routes with Swagger/Postman export.

### Option B — Global path-rewrite hook, official

Ship `TenantRouteMiddleware` that the consumer installs via `app.use()` in their `main.ts`. Library exports a `createTenantRouteMiddleware(config)` helper. `ResolveOrganizationMiddleware` continues to do member-check AFTER the rewrite.

**Pros:** small change, no reflection magic.
**Cons:** still requires consumer to wire it up — not "automatic".

### Option C — Switch to a request-scoped resolver

Instead of prefix-based routing, use header-based tenancy: a `X-Organization-Slug` header. `ResolveOrganizationMiddleware` reads that header, no URL rewrite needed. Breaks API URL convention parity with Laravel.

**Recommendation:** Option A. It's what the config shape promises. Failing that, ship Option B as an official workaround and document the trade-off.

## Acceptance criteria for fix

- [ ] `RhinoModule.forRoot({routeGroups: { tenant: { prefix: ':organization', models: '*' } }})` makes `GET /api/acme/projects` reach `GlobalController.index` with `req.organization = { id, slug: 'acme' }` set.
- [ ] Route group's `middleware` array fires for tenant routes.
- [ ] `skipAuth: true` groups bypass JwtAuthGuard.
- [ ] Existing `/api/:modelSlug` direct calls still work when no route group declares them (or are rejected deterministically — document either).
- [ ] Postman / TypeScript export reflect the resolved prefix.
- [ ] E2E test in the library: register 2 route groups (`tenant` with `:organization`, `public` with `public`), hit routes under each, verify middleware runs and controller sees right `req` shape.
- [ ] Remove the workaround in `rhino-template/server-nestjs/src/main.ts`.
