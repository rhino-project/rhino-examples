# BP-003 — Array-form `create_fields` emits everything as `.optional()` — required fields ignored

**Type:** bug
**Severity:** medium — every model needs a hand-authored validation override
**Affects:** `@rhino-dev/rhino-nestjs@0.1.0`

## Symptom

YAML:

```yaml
permissions:
  admin:
    actions: [index, show, store, update, destroy]
    show_fields: "*"
    create_fields:
      - title            # supposed to be required
      - description
      - status
    update_fields:
      - title
```

Generated `ProjectResource.ts`:

```ts
const validationStore: Record<string, z.ZodTypeAny> = {
  admin: z.object({
    title: z.string().optional(),         // ← optional, not required
    description: z.string().optional(),
    status: z.string().optional(),
  }),
  ...
};
```

Runtime result: a POST with empty body returns 201 and writes `{ title: null }` to the DB (or crashes at Prisma layer because `title` is non-nullable).

## Root cause

`src/blueprint/generators/resource-definition-generator.ts` → `zodTypeForColumn(fieldName, columns, presence)`:

When `create_fields` is declared as an ARRAY (no presence modifiers), `presence` arrives as `'optional'` (the hard-coded fallback in `buildZodSchema()` → `zodTypeForColumn(f, columns, 'optional')`).

```ts
switch (presence) {
  case 'required':
    return baseType;
  case 'nullable':
    return `${baseType}.nullable().optional()`;
  case 'sometimes':
  case 'optional':
    return `${baseType}.optional()`;   // ← wins here
}
```

The array form has no way to express required-ness. The object form `{ title: required, description: nullable }` works correctly.

## Workaround

In `src/rhino.config.ts` of the template, every model declares an explicit `validation:` base schema that marks correct requiredness, then spread-overrides the generated `validationStore`/`validationUpdate`.

```ts
const projectStore = z.object({
  title: z.string().min(1),
  description: z.string().optional(),
  // ...
});

const projectUpdate = projectStore.partial();

projects: {
  ...projectsRegistration,
  validation: projectStore,
  validationStore: {
    owner: projectStore,
    admin: projectStore,
    manager: projectStore.omit({ budget: true, internalNotes: true }),
    '*': z.object({}),
  },
  validationUpdate: { ... },
}
```

Double the code — the blueprint stopped carrying its weight for validation.

## Proposed fix

Extend the YAML schema to allow mixed-form `create_fields`:

```yaml
create_fields:
  title:      required
  description: nullable
  status:      nullable
```

OR allow per-column required flag on the COLUMN itself (already implied by `nullable: false`):

```yaml
columns:
  title:
    type: string
    nullable: false   # ← generator already sees this

create_fields: [title, description, status]
```

The generator can then: `if (col.nullable === false && presence === 'optional') return baseType;`

Essentially, mirror the Laravel generator which reads the column's `nullable:` flag to decide requiredness when presence isn't specified.

## Acceptance criteria for fix

- [ ] `columns.title.nullable: false` + `create_fields: [title]` (array form) generates `title: z.string()` (required).
- [ ] `columns.description.nullable: true` + `create_fields: [description]` generates `description: z.string().nullable().optional()`.
- [ ] Object form `{ title: required }` still overrides column nullable (explicit wins).
- [ ] New test in `resource-definition-generator.spec.ts`: array-form with mixed nullable columns → correct Zod types.
- [ ] Template can delete the hand-authored `projectStore` / `taskStore` / etc. in `rhino.config.ts`.
